# Revision Guide

This repo has 1277 Java files spread across ~250 folders, organized by *source*
(Neetcode150, Striver, TakeUForward, HitBullseye, getSDEReady, Practice2025...)
rather than by *pattern*. That's fine for solving problems, but it makes it
hard to answer "what should I revise today" after a few months away.

`progress.csv` (repo root) is a flat index of every problem file, tagged with
a best-guess **pattern** (Two Pointers, Sliding Window, DP, Graph, etc.),
generated by scanning folder/file names — not by reading every file's logic.
It will mis-tag a few files and leaves ~100 as `Uncategorized`; fix tags as
you encounter them rather than trying to perfect it up front.

The CSV intentionally does **not** carry fake "last touched" dates from git
history — this repo's commit history is a handful of bulk "add everything"
commits, so per-file commit dates would be meaningless. Instead there are
three blank columns for you to fill in as you go:

- **Last Reviewed (YYYY-MM-DD)** — the date you last solved/re-solved it
- **Confidence (1-5)** — how sure you are you could solve it cold today
- **Notes** — the trick/insight you keep forgetting

## How to revise after a long break

1. Open `progress.csv` in Numbers/Excel/Sheets, sort by **Pattern**.
2. For each pattern, pick 2-3 problems with the oldest "Last Reviewed" date
   (or blank, if never logged) and re-solve them cold, no looking at your
   old code first.
3. Update **Last Reviewed** and **Confidence** immediately after.
4. Spend most of your time on patterns where every problem is still at
   Confidence 1-2 — that's a sign the underlying technique never solidified,
   not that you need more problems in that bucket.
5. Don't try to revise everything in one sitting — rotate patterns weekly.
   A pattern you rated 4-5 a month ago needs a much lighter touch than one
   you've never logged.

## Keeping the index current

When you add a new problem file, append a row to `progress.csv` by hand
(Pattern, Problem, Source, Path) — the regeneration script isn't meant to be
rerun on every commit, since it would wipe your Last Reviewed/Confidence/Notes
columns. Only rerun it if you want a fresh full rescan; back up your filled-in
columns first if so.
